# 팀 운영 가이드

팀으로 QA 자동화를 운영하는 방법입니다.

## 브랜치 전략

### 기본 브랜치

- `main` - 안정된 테스트만 포함
- `develop` - 개발 중인 테스트

### 작업 브랜치

```bash
# 새 테스트 작성 시
git checkout -b test/로그인-실패-케이스

# 버그 수정 시
git checkout -b fix/결제-셀렉터-수정
```

### 브랜치 네이밍

| 접두사 | 용도 | 예시 |
|--------|------|------|
| `test/` | 새 테스트 추가 | `test/회원가입-검증` |
| `fix/` | 테스트 수정 | `fix/로그인-타임아웃` |
| `refactor/` | 구조 개선 | `refactor/page-object-정리` |

## 워크플로우

### 1. 새 테스트 작성

```bash
# 1. 브랜치 생성
git checkout -b test/새로운-기능

# 2. 테스트 작성 (Claude 사용)
claude

# 3. 커밋
git add .
git commit -m "test: 새로운 기능 테스트 추가"

# 4. 푸시
git push -u origin test/새로운-기능

# 5. PR 생성
```

### 2. Pull Request 작성

PR 템플릿:

```markdown
## 테스트 설명
- 어떤 기능을 테스트하는지

## 테스트 케이스
- [ ] 정상 케이스
- [ ] 에러 케이스

## 실행 결과
- 로컬에서 테스트 통과 확인

## 스크린샷
(필요시 첨부)
```

### 3. 코드 리뷰

리뷰어 체크리스트:

- [ ] 테스트 이름이 명확한가?
- [ ] Page Object를 활용했는가?
- [ ] 하드코딩된 값이 없는가?
- [ ] 다른 테스트에 영향을 주지 않는가?

## 테스트 분류

### 폴더 구조

```
tests/
├── auth/           # 인증 (로그인, 회원가입, 비밀번호)
├── user/           # 사용자 (프로필, 설정)
├── product/        # 상품 (목록, 상세, 검색)
├── checkout/       # 결제 (장바구니, 주문)
└── admin/          # 관리자
```

### 담당자 분배

| 영역 | 담당자 | 테스트 수 |
|------|--------|----------|
| auth | QA1 | 15개 |
| checkout | QA2 | 20개 |
| product | QA3 | 25개 |

## 공유 방법

### 1. 새 테스트 공유

Slack/Teams에 공유:

```
[새 테스트 추가]
- 영역: 결제
- 파일: tests/checkout/refund.spec.ts
- 케이스: 환불 요청 테스트
- PR: #123
```

### 2. 주간 리뷰

매주 테스트 현황 공유:

- 신규 테스트 수
- 실패율 변화
- 주요 이슈

## CI/CD 연동

### GitHub Actions

테스트가 자동 실행되는 시점:

- PR 생성 시
- main 브랜치 머지 시
- 매일 자정 (스케줄)

### 실패 알림

테스트 실패 시:
1. GitHub에서 이메일 알림
2. Slack 채널에 알림 (설정 시)

## 테스트 유지보수

### 정기 점검

- **매주**: 실패하는 테스트 확인
- **매월**: 불필요한 테스트 정리
- **분기**: 테스트 커버리지 검토

### 실패 대응

```
1. 실패 원인 파악
   - 테스트 버그? → 테스트 수정
   - 실제 버그? → 개발팀에 리포트

2. 임시 스킵 (급한 경우)
   test.skip('임시 스킵: 이유', ...)

3. 수정 후 스킵 해제
```

## 신규 팀원 온보딩

4. 코드 리뷰 받기
